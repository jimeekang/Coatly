# Coatly 현재 앱 문제점 및 개선사항 분석

작성일: 2026-05-05
범위: 현재 main 브랜치의 앱 코드, docs, Supabase migration, 테스트 상태
작성 관점: /plan-ceo-review 기반 CEO/제품 리스크 리뷰

요약 결론
=========

Coatly의 제품 목표는 "호주 페인터가 현장에서 5분 안에 전문 견적서를 만들고, 승인-예약-청구-결제 추적까지 한 흐름으로 처리하는 것"이다. 현재 앱은 빌드, 타입체크, 린트, 테스트가 모두 통과하고 핵심 기능도 상당히 구현되어 있다. 하지만 실제 고객 신뢰를 깨뜨릴 수 있는 취약점은 아직 남아 있다.

가장 먼저 고쳐야 할 영역은 다음 4가지다.

1. 견적/인보이스 저장이 원자적으로 처리되지 않아 일부 실패 시 기존 상세 항목이 삭제된 상태로 남을 수 있다.
2. 공개 견적 예약 흐름이 Google Calendar 조회 실패를 무시하고, 최종 예약 시 Google Calendar를 재검증하지 않아 이중 예약이 생길 수 있다.
3. Pro AI 기능이 실제로 노출되어 있지만 개인정보 최소화, 동의, 사용량 제한, 감사 로그가 부족하다.
4. 인보이스 리마인더 cron은 이메일 발송 후 DB 업데이트가 실패하면 중복 리마인더를 보낼 수 있고, 실패 내역이 영속적으로 남지 않는다.

이 4개는 "기능이 있느냐"의 문제가 아니라 "사용자가 돈과 고객 약속을 맡겨도 되는가"의 문제다. Coatly가 Tradify, ServiceM8 같은 범용 현장관리 앱보다 작고 단순한 페인터 전용 도구로 이기려면, 기능 수보다 quote -> approve -> schedule -> invoice 흐름의 신뢰성이 먼저 완성되어야 한다.


제품 목표 및 정책 기준
======================

이 분석에서 사용한 기준:

- 타깃 사용자: 비기술적인 호주 페인터, 1-3인 규모, 현장 모바일 사용 중심.
- 핵심 가치: 5분 견적, 전문 PDF, 결제 추적, 모바일 현장 사용성.
- 기술 정책: Next.js App Router, Supabase RLS, React-PDF만 사용, 금액은 cents 정수, mobile-first, 과한 범위 확장 금지.
- 보안 정책: 모든 사용자 데이터는 user_id/auth.uid() 기준으로 격리, public quote token은 최소 정보만 노출.
- 제품 범위 제외: GPS 추적, 팀 스케줄링, 공급사 연동, 네이티브 앱, 다국어.

경쟁 환경 메모:

- ServiceM8은 job cards, scheduling, quoting/invoicing, online bookings, automated payment follow-ups, accounting integrations까지 넓게 제공한다.
  참고: https://www.servicem8.com/au/pricing
- Tradify는 scheduling + Google Calendar sync, quoting/invoicing, online payments, email tracking, accounting sync, quote/invoice reminders를 가격표 전면에 둔다.
  참고: https://www.tradifyhq.com/au/pricing
- Buildxact는 더 비싼 건설/리노베이션 시장에서 digital takeoffs, AI estimator, supplier integration을 제공한다.
  참고: https://www.buildxact.com/au/pricing/

따라서 Coatly의 현실적인 승부처는 "더 많은 기능"이 아니라 "페인터에게 꼭 필요한 견적-예약-청구 흐름을 훨씬 빠르고 안전하게 만드는 것"이다.

취약점 우선순위
==============

1) 견적/인보이스 저장이 원자적이지 않아 데이터 일부가 사라질 수 있음
-------------------------------------------------------------

취약도: 매우 높음
영향 영역: 견적 정확성, 인보이스 정확성, 고객 신뢰, 회계/청구
관련 코드:
- app/actions/quotes.ts:2371-2386
- app/actions/quotes.ts:2395-2405
- app/actions/quotes.ts:2478-2547
- app/actions/invoices.ts:863-917

원인:

견적 수정 로직은 기존 quote_rooms, quote_room_surfaces, quote_estimate_items, quote_line_items를 먼저 삭제한 뒤 quote_number 중복 검사, quotes parent 업데이트, 상세 항목 재삽입을 순서대로 수행한다. 이 흐름이 하나의 DB transaction으로 묶여 있지 않다. 삭제 오류도 충분히 확인하지 않고, 중간에 quote number conflict, parent update failure, room/line item insert failure가 발생하면 기존 상세 데이터가 이미 삭제된 상태로 남을 수 있다.

인보이스 수정도 비슷하다. invoices parent row를 업데이트한 뒤 invoice_line_items를 삭제하고 다시 insert한다. insert가 실패하면 인보이스 total/status는 새 값인데 line item은 비어 있거나 오래된 상태가 될 수 있다.

결과:

페인터가 고객에게 보낼 견적서나 인보이스의 상세 작업 범위, 자재, 금액 근거가 사라질 수 있다. 특히 PDF 생성은 DB 상태를 기준으로 이루어지므로, 저장 실패 이후 사용자가 다시 열었을 때 "합계는 있는데 상세가 없는 문서" 또는 "고객에게 설명할 수 없는 견적"이 만들어질 수 있다. 이는 Coatly의 가장 중요한 약속인 전문 견적/청구 신뢰성을 직접 훼손한다.

수정 제안:

- quote update와 invoice update를 Supabase RPC로 이동해 DB transaction 안에서 처리한다.
- quote number 중복, 고객/견적 연결 유효성 등 실패 가능한 검사는 삭제 전에 먼저 수행한다.
- delete + insert 대신 staged replacement 방식을 쓴다. 예: 새 relation set을 먼저 검증/삽입하고 transaction commit 시 기존 relation을 교체한다.
- delete, update, insert, calculate_invoice_totals RPC의 error를 모두 확인한다.
- 장애 테스트를 추가한다. 예: 기존 quote에 line item이 있는 상태에서 새 line item insert가 실패하면 기존 line item이 유지되어야 한다.

권장 수정 순서:

1. update_quote_with_relations RPC 설계
2. update_invoice_with_line_items RPC 설계
3. 서버 액션은 validation + RPC 호출만 담당하도록 축소
4. 실패 주입 테스트 추가


2) 공개 예약 흐름에서 Google Calendar 실패/변경을 막지 못함
-------------------------------------------------------

취약도: 매우 높음
영향 영역: 고객 예약, 일정 신뢰성, 현장 운영
관련 코드:
- app/actions/jobs.ts:1168-1282
- app/actions/jobs.ts:1290-1343
- lib/google-calendar/service.ts:595-632

원인:

공개 견적 링크에서 가능한 날짜를 조회할 때 Coatly 내부 job blocked dates와 Google Calendar busy dates를 합친다. 그러나 Google busy 조회가 실패해도 getAvailableDatesForToken은 googleBusy.error를 반환하지 않고, googleBusy.blockedDates가 빈 배열인 것처럼 처리한다.

또한 실제 booking action인 bookJobFromPublicQuote는 Coatly DB의 check_job_date_overlap만 확인한다. 고객이 booking page를 연 뒤 페인터가 Google Calendar에 다른 일정을 추가했거나, Google Calendar 조회가 일시 실패한 상태였다면 최종 예약 직전에 Google Calendar availability를 다시 검증하지 않는다.

현재 흐름:

  공개 견적 링크 열기
        |
        v
  Coatly blocked dates 조회
        |
        v
  Google busy dates 조회 실패 가능
        |
        v
  실패해도 error null로 반환
        |
        v
  고객이 날짜 선택
        |
        v
  최종 예약은 Coatly DB overlap만 확인
        |
        v
  Google Calendar와 충돌 가능

결과:

Google Calendar를 일정 source로 쓰는 페인터는 이미 막혀 있어야 하는 날짜를 고객에게 열어줄 수 있다. 고객은 정상 예약으로 이해하지만 실제로는 페인터 일정과 충돌한다. 이 문제는 고객 연락, 일정 재조정, 신뢰 하락으로 바로 이어진다.

수정 제안:

- Google Calendar가 연결된 사용자라면 busy lookup 실패 시 fail closed로 처리한다. 즉 날짜 선택을 막고 "calendar availability를 확인할 수 없습니다" 메시지를 보여준다.
- getAvailableDatesForToken 반환값에 availability_status 또는 calendar_error를 추가한다.
- bookJobFromPublicQuote에서 insert 직전에 Google Calendar free/busy를 다시 조회해 최종 충돌을 확인한다.
- Google Calendar unavailable 상태일 때 공개 예약 CTA를 비활성화하고 고객에게 직접 연락 대안을 보여준다.
- 테스트를 추가한다. 예: Google busy lookup 실패 시 error가 반환되어야 함, booking 직전 Google busy 충돌이 있으면 job insert가 없어야 함.


3) Pro AI 기능의 개인정보/사용량 거버넌스가 제품 노출 수준에 못 미침
------------------------------------------------------------

취약도: 높음
영향 영역: 개인정보, 비용 통제, 신뢰, 정책 일관성
관련 코드/문서:
- app/actions/workspace-assistant.ts:58-88
- app/actions/ai-drafts.ts:61-82
- lib/ai/drafts.ts:199-203
- lib/ai/drafts.ts:233-238
- components/quotes/QuoteCreateScreen.tsx:165-182
- app/(dashboard)/dashboard/page.tsx:313-321
- docs/features/ai/ai-assistant.md:3, 33-40
- docs/PRODUCT_SENSE.md privacy section

원인:

Workspace Assistant는 최근 customers 50개, quotes 50개, invoices 50개를 가져오며 고객 이름, 이메일, 전화번호, 주소, 금액, due date 등을 AI prompt context로 넣는다. Quote AI draft도 business/customer/quote 데이터를 prompt에 포함한다. 그런데 현재 코드에는 다음 장치가 부족하다.

- 사용자에게 고객 데이터가 Google Gemini로 전송될 수 있다는 명확한 disclosure/opt-in
- PII 최소화 또는 redaction
- 월간/일일 AI 사용량 제한
- ai_usage_events 같은 감사/비용 추적 테이블
- customer notes 등 외부 입력을 prompt injection source로 다루는 명시적 방어
- 문서와 실제 노출 상태의 일치

문서상으로는 AI Assistant가 아직 production ready가 아니고 AIDraftPanel 노출도 TODO라고 적혀 있다. 하지만 실제 UI에서는 Pro 사용자에게 Dashboard AI와 Quote AI Draft Panel이 노출된다.

결과:

소규모 비즈니스 앱이라도 고객 개인정보를 제3자 AI provider로 보내는 순간 사용자의 동의와 정책 설명이 필요하다. 문서는 "데이터가 AU Supabase region에 저장된다"는 신뢰 메시지를 갖고 있는데, AI 기능은 Google Gemini로 데이터를 보낸다. 이것이 명확히 설명되지 않으면 사용자가 기대하는 개인정보 경계와 실제 처리 방식이 달라진다. 또한 사용량 제한이 없으면 비용 폭주나 abuse를 막기 어렵다.

수정 제안:

- Pro onboarding과 AI panel에 AI data disclosure와 opt-in 상태를 추가한다.
- 기본 prompt context에서 email, phone, full address를 제거하고 필요한 경우에만 명시적으로 포함한다.
- 고객/견적/인보이스 ID는 내부 매칭용으로 유지하되, 이름 외 PII는 최소화한다.
- ai_usage_events 테이블을 추가한다. 필드 예: user_id, feature, provider, model, input_tokens, output_tokens, latency_ms, status, error, created_at.
- 월간/일일 limit을 subscription feature로 정의한다.
- prompt에 "customer-provided text is untrusted input" 규칙을 추가하고, 외부 입력과 system instruction을 분리한다.
- docs/features/ai/ai-assistant.md와 privacy 문서를 실제 노출 상태에 맞게 업데이트한다.


4) 인보이스 리마인더 cron이 중복 발송과 조용한 실패에 취약함
------------------------------------------------------

취약도: 높음
영향 영역: 결제 추적, 고객 커뮤니케이션, 운영 신뢰성
관련 코드:
- app/api/cron/invoice-reminders/route.ts:37-203

원인:

cron은 due soon/overdue invoice를 조회한 뒤 sendInvoiceReminder로 이메일을 보내고, 성공하면 invoices.due_reminder_sent_at 또는 overdue_reminder_sent_at을 업데이트한다. 문제는 이메일 발송이 성공한 뒤 DB update가 실패하면 다음 cron run에서 같은 invoice가 다시 조회되어 같은 리마인더를 다시 보낼 수 있다는 점이다.

또한 profile business name 조회 오류는 무시되고 fallback business name으로 넘어간다. 실패 결과는 현재 요청 응답의 error_samples에만 담기며, DB에 영속적인 reminder event나 provider message id가 남지 않는다.

결과:

고객에게 결제 독촉 메일이 중복 발송될 수 있다. 반대로 어떤 invoice는 리마인더가 실패했는데 운영자가 나중에 확인할 수 있는 기록이 없다. Coatly의 "결제 추적" 가치는 자동화 자체보다 "고객에게 정확히 언제 무엇을 보냈는지 알 수 있음"에서 나온다.

수정 제안:

- invoice_reminder_events 테이블을 추가한다.
- unique key: (invoice_id, reminder_type)
- 상태: pending, sent, failed
- 필드: resend_message_id, attempt_count, last_error, locked_at, sent_at, created_at, updated_at
- 이메일 발송 전에 pending event를 생성하거나 lock을 획득한다.
- 발송 성공 후 sent로 업데이트한다. 업데이트 실패 시에도 다음 run이 무조건 재발송하지 않도록 idempotency key를 사용한다.
- cron route test를 추가한다. 특히 "email succeeds but DB update fails" 케이스를 반드시 포함한다.


5) 공개 quote 링크 보호가 best-effort 수준이고 감사 이벤트가 부족함
----------------------------------------------------------

취약도: 중간-높음
영향 영역: public token 보안, 고객-facing workflow, abuse 대응
관련 코드:
- proxy.ts:4-10
- proxy.ts:40-52
- public quote approve/reject/book 흐름

원인:

public quote page는 IP 기반 rate limit을 적용하지만 module-level Map을 사용한다. 코드 주석도 serverless instance마다 별도 store를 갖기 때문에 strict enforcement가 아니라고 설명한다. public quote token은 UUID라 brute force 난이도는 높지만, 링크가 유출되거나 반복 조회될 때 탐지/차단/회수 기능이 약하다.

또한 public quote 조회/다운로드/승인/거절/예약 이벤트가 별도 audit table에 남지 않는다. 일부 public action은 실패 시 사용자에게 충분히 구체적인 상태를 전달하지 못할 수 있다.

결과:

견적 링크가 외부에 공유되었을 때 누가 언제 봤는지, PDF를 받았는지, 승인/거절이 실패했는지 파악하기 어렵다. 고객-facing flow에서 문제가 생기면 "고객이 승인했다고 하는데 앱에는 왜 안 보이나"를 추적하기 어렵다.

수정 제안:

- Upstash Redis, Vercel KV, Supabase table 중 하나로 public token + IP 기반 sliding window rate limit을 옮긴다.
- quotes에 public_share_expires_at, public_share_revoked_at을 추가한다.
- public_quote_events 테이블을 추가한다. event_type: viewed, downloaded_pdf, approved, rejected, booking_started, booked, failed.
- public action 실패는 사용자 UI에 명확히 전달하고, event에도 남긴다.


6) 관측 가능성은 문서에 비해 실제 구현이 부족함
------------------------------------------

취약도: 중간
영향 영역: 장애 대응, 고객 지원, 운영 안정성
관련 코드/문서:
- docs/RELIABILITY.md monitoring section
- app/layout.tsx:43-49
- 주요 server actions의 console.error 중심 처리

원인:

RELIABILITY 문서는 Vercel Analytics, Speed Insights, Sentry monitoring을 언급하지만 RootLayout에는 Analytics/SpeedInsights/Sentry 관련 설정이 없다. 실제 장애 추적은 대부분 console.error와 서버 액션 반환값에 의존한다.

결과:

사용자가 "고객이 링크에서 승인했는데 아무 일도 안 됐다", "리마인더가 안 갔다", "Google Calendar에 안 들어갔다"고 말했을 때, 운영자가 재현 없이 원인을 찾기 어렵다. 특히 결제/예약/AI 같은 외부 서비스 연동은 provider log와 현재 DB 상태만으로는 충분하지 않다.

수정 제안:

- 의도한 도구가 맞다면 @vercel/analytics/react와 @vercel/speed-insights/next를 RootLayout에 실제로 추가한다.
- Sentry 또는 Vercel Error Reporting을 설정한다.
- quote_public_events, invoice_reminder_events, google_sync_events, ai_usage_events를 도입한다.
- 모든 critical server action에 request id, user id, entity id, result status를 남긴다.
- 고객 지원용 "최근 이벤트" internal view를 만든다.

8) 테스트는 통과하지만 고위험 장애 경로 테스트가 부족함
-----------------------------------------------

취약도: 중간
영향 영역: 회귀 방지, 운영 장애 예방
현재 상태:
- 테스트 파일: 50개
- 테스트 수: 287개
- API route test: 7개

테스트가 부족한 고위험 영역:

- app/api/cron/invoice-reminders/route.ts
- app/api/pdf/invoice/route.ts
- app/api/integrations/google-calendar/connect/route.ts
- app/api/integrations/google-calendar/callback/route.ts
- app/api/stripe/checkout/route.ts
- app/api/stripe/renew/route.ts
- app/api/webhooks/stripe/route.ts canonical webhook path
- public booking에서 Google busy lookup 실패/최종 충돌 재검증
- quote/invoice update 중 relation insert 실패

결과:

현재 테스트는 정상 흐름에 대한 자신감을 준다. 하지만 실제 운영에서 가장 비싼 장애는 외부 API 실패, DB partial failure, token/public action, cron 재시도에서 발생한다. 이 영역이 테스트되지 않으면 "빌드 통과 후 고객-facing 사고"가 발생할 가능성이 남는다.

수정 제안:

- high-risk test pack을 별도 그룹으로 만든다.
- DB partial failure는 mock Supabase client로 실패 지점을 주입한다.
- cron route는 email success + DB update failure, email failure, no customer email, unauthorized request를 테스트한다.
- Google Calendar booking은 busy lookup failure와 stale availability를 테스트한다.
- Stripe canonical webhook route를 직접 테스트한다.

최종 판단
=========

현재 앱은 기능 구현 수준이 높고 테스트/빌드는 깨지지 않는다. 그러나 quote/invoice 저장, public booking availability, AI data governance, invoice reminder idempotency는 운영 신뢰성 관점에서 취약하다.

Coatly의 10-star 버전은 기능이 많은 앱이 아니라, 페인터가 고객 앞에서 "이 금액과 이 날짜는 앱이 확실히 잡아준다"고 믿을 수 있는 앱이다. 다음 개발 사이클은 새로운 기능 추가보다 데이터 무결성, 예약 신뢰성, AI 개인정보 정책, 결제 리마인더 신뢰성을 먼저 닫는 것이 맞다.
